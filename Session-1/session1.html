🎓 Theory Block — The Shift-Left Security Mindset

🔗 Bridge from Your QA World
You've spent 9 years catching bugs early. You know that a bug found in production costs 10–100× more to fix than one caught in a unit test. DevSecOps applies this exact principle to security vulnerabilities. A vulnerability found in code review = 1 hour to fix. The same vulnerability found after a GC breach = months of incident response, public hearings, and career-defining consequences.
🔍 What IS DevSecOps?

DevSecOps = Development + Security + Operations

It's the practice of integrating security as a shared responsibility across the entire software development lifecycle — not as a gate at the end, but as a continuous thread woven through every stage.

⚠️ The Old World (Pre-DevSecOps)
Dev builds for 6 months → throws code over the wall to Ops → Security does a pentest 2 weeks before launch → Security says "there are 47 critical vulnerabilities" → Dev says "we can't delay the launch" → Security says "okay but sign off" → vulnerability ships to production → breach happens 8 months later. Sound familiar?
✅ The DevSecOps World
Security is automated into every pipeline stage. SAST scans on every commit. Dependency checks on every build. Container image scanning before every deployment. Security gates that block bad code — automatically, just like your quality gates block failing tests.

🇨🇦 Why Does the Government of Canada Care?

The GC is the largest employer in Canada and manages some of the most sensitive data in the country — tax records (CRA), immigration files (IRCC), military intelligence (DND), health data (HC), and more. A breach isn't just an embarrassment — it's a national security event.

🔴 Real GC Breach Context
In 2020, the GCKey system (used by millions of Canadians to access government services) was breached via credential stuffing. ~9,000 accounts compromised. ESDC and CERB applications were fraudulently accessed. This happened because security wasn't built into the authentication pipeline — it was bolted on. DevSecOps practices like automated security testing and rate-limiting would have caught this earlier.

The GC Security Mandate

The Treasury Board Secretariat (TBS) now mandates DevSecOps practices through:

  • 📜ITSG-33 — IT Security Risk Management: A Lifecycle Approach — the GC's security bible
  • 📜PBMM — Protected B, Medium Integrity, Medium Availability — the cloud security baseline
  • 📜GC Zero Trust Strategy (2023) — "never trust, always verify" across all GC systems
  • 📜CCCS Cloud Medium Profile — 63 controls for cloud workloads hosting GC data

A federal security auditor will ask you: "How do you map your pipeline security controls to ITSG-33 safeguards?" — You'll be able to answer this by Month 3.

🎯 Lab Scenario (Federal Context)
You're a DevSecOps engineer at a GC department. A developer just pushed code to a new repository. Before this goes anywhere near a pipeline, you need to: (1) scan it for secrets/credentials, (2) scan it for vulnerable dependencies, and (3) run a SAST scan. You'll do all 3 using free tools today.

🔬 Lab Steps
1

Create a "vulnerable" test repository

Create a directory and add an intentionally insecure Python file to simulate a GC developer's mistake:

mkdir gc-devsecops-lab && cd gc-devsecops-lab
git init

# Create a "vulnerable" Python file
cat > app.py << 'EOF'
import boto3
import requests

# BAD: Hardcoded credentials (a real federal nightmare)
AWS_ACCESS_KEY = "AKIAIOSFODNN7EXAMPLE"
AWS_SECRET_KEY = "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"
DB_PASSWORD = "SuperSecret123!"

def get_user_data(user_id):
    # BAD: SQL injection vulnerability
    query = "SELECT * FROM users WHERE id = " + user_id
    return query

def fetch_data(url):
    # BAD: No SSL verification (ITSG-33 violation)
    return requests.get(url, verify=False)
EOF

git add . && git commit -m "initial commit"
2

Install and run Semgrep (SAST Scan)

# Install Semgrep
pip install semgrep

# Run a SAST scan against the Python file
semgrep --config=auto app.py

# For a GC-relevant ruleset (OWASP Top 10):
semgrep --config=p/owasp-top-ten app.py

📋 Expected output: Semgrep will flag the SQL injection pattern and the disabled SSL verification as security issues.

3

Run a Secret Scan with truffleHog

# Install truffleHog
pip install trufflehog3

# Scan the local repo for secrets
trufflehog3 filesystem . --output report.json

# Or use the simpler git-secrets approach:
pip install detect-secrets
detect-secrets scan > .secrets.baseline
detect-secrets audit .secrets.baseline

📋 Expected output: The tool will find the AWS_ACCESS_KEY and DB_PASSWORD as high-entropy strings / known secret patterns.

4

Create a requirements.txt with a vulnerable library

# Add a requirements file with a known-vulnerable package
cat > requirements.txt << 'EOF'
requests==2.18.4
flask==0.12.2
pyyaml==3.13
EOF

# Run SCA scan
pip install pip-audit
pip-audit -r requirements.txt

📋 Expected output: pip-audit will flag CVEs in flask 0.12.2 and pyyaml 3.13 — these are real known vulnerabilities.

5

Document your findings like a real GC security engineer

Create a simple findings report:

cat > SECURITY_FINDINGS.md << 'EOF'

## Date: [Today's Date]
## Engineer: Parinder

## CRITICAL FINDINGS
| ID | Type | File | Issue | ITSG-33 Ref |
|----|------|------|-------|-------------|
| F-001 | Secret | app.py:6 | AWS Access Key hardcoded | AC-2 |
| F-002 | Secret | app.py:7 | AWS Secret Key hardcoded | AC-2 |

| F-004 | SAST | app.py:17 | SSL verification disabled | SC-8 |

🃏 Flashcards — Session 1

Click any card to reveal the answer. Master these before your next session.

Core Concept
What does "Shift-Left Security" mean?
👆 Click to flip
Moving security testing and checks as early as possible in the SDLC — into the coding and build stages rather than just before or after deployment. Like catching bugs in unit tests instead of production.
Acronym
What is SAST and when does it run?
👆 Click to flip
Static Application Security Testing. Scans source code WITHOUT running it. Runs at the CODE or BUILD stage of the pipeline. Tools: Semgrep, SonarQube, Checkmarx. Catches injection flaws, hardcoded secrets, insecure patterns.
Acronym
What is DAST and when does it run?
👆 Click to flip
Dynamic Application Security Testing. Tests a RUNNING application by simulating attacks. Runs at the TEST stage. Tools: OWASP ZAP, Burp Suite. Catches runtime vulnerabilities SAST can't see.
Acronym
What is SCA and what real incident is it famous for?
👆 Click to flip
Software Composition Analysis — scans open-source dependencies for known CVEs. Famous for Log4Shell (CVE-2021-44228): Log4j vulnerability affected thousands of GC systems. SCA would have flagged the vulnerable version before deployment.
GC Compliance
What does ITSG-33 stand for and who publishes it?
👆 Click to flip
IT Security Guidance 33 — "IT Security Risk Management: A Lifecycle Approach." Published by the Communications Security Establishment (CSE). It's the GC's primary framework for managing security risks throughout the SDLC.
GC Compliance
What does PBMM stand for and what cloud does it apply to?
👆 Click to flip
Protected B, Medium Integrity, Medium Availability. The security baseline for GC cloud workloads handling sensitive (not secret) data. Azure has a PBMM blueprint. This is what most GC departments use for their cloud environments.
3 Pillars
What are the 3 pillars of DevSecOps?
👆 Click to flip
1. 👥 PEOPLE — Security is a shared responsibility across Dev, Sec, and Ops teams.
2. ⚙️ PROCESS — Security baked into SDLC, change management, IR processes.
3. 🔧 TECHNOLOGY — Automated security tools in every pipeline stage.
GC Context
What is SA&A in a GC context?
👆 Click to flip
Security Assessment & Authorization — the GC's equivalent of ATO (Authorization to Operate). Every GC production system must go through SA&A before launch. DevSecOps helps by generating automated evidence (scan reports, compliance checks) for the SA&A package.
Your QA Bridge
What is the DevSecOps equivalent of a quality gate?
👆 Click to flip
A SECURITY GATE — a pipeline checkpoint that automatically blocks a build/deployment if security thresholds aren't met. Example: "Block deployment if SAST finds any Critical severity issue." Same concept as blocking a build with failing test coverage.
Real Incident
What was the SolarWinds attack and why does it matter for DevSecOps?
👆 Click to flip
2020 supply chain attack: hackers inserted malicious code into SolarWinds Orion build pipeline. Affected 18,000+ organizations including US gov agencies. Lesson: DevSecOps must secure the BUILD PIPELINE itself — not just the application code. GC now requires supply chain security controls.
Acronym
What is an SBOM and why does TBS care?
👆 Click to flip
Software Bill of Materials — a complete inventory of all software components and their versions in an application. TBS recommends SBOMs for GC software so that when a new CVE drops (like Log4Shell), they can instantly know which systems are affected.
Interview Trap
"Isn't DevSecOps just DevOps with security?" — How do you answer this?
👆 Click to flip
No. DevOps treats security as a separate phase (or ignores it). DevSecOps embeds security as a continuous, automated, shared responsibility throughout EVERY stage. The culture shift is as important as the tools — everyone owns security, not just the security team.

Question 1 of 5 — Conceptual
In the traditional "waterfall" security model, when does the security team typically review the application?
✅ Correct. In the old model, security was a gate at the END — often called "bolt-on security." This is why breaches happened at production. DevSecOps fixes this by embedding security from the very first commit.
Question 2 of 5 — Conceptual
Which type of security scan would catch the Log4Shell vulnerability (CVE-2021-44228) in a Java application's dependencies?
✅ Correct. Log4Shell was a vulnerability IN a dependency (log4j-core). SCA tools like Snyk, Dependabot, and pip-audit scan your dependency files against the CVE database. This is exactly why SCA is mandatory in any GC-compliant pipeline.
Question 3 of 5 — Practical Scenario
A developer on your GC team commits code with a database password hardcoded in a Python file and pushes to the main branch. As a DevSecOps engineer, which tool should have caught this BEFORE it reached main?
✅ Correct. Secret scanning runs as a pre-commit hook or pipeline step on EVERY commit and looks for patterns like passwords, API keys, and tokens. This is a critical GC requirement — secrets in code have caused real federal data breaches. Tools: detect-secrets, GitGuardian, Azure Defender for DevOps.
Question 4 of 5 — Practical Scenario
Your team is building a CI/CD pipeline for a GC application handling Protected B data. You want to "shift left" as aggressively as possible. Which ordering of security checks is MOST correct?
✅ Correct. The "most left" checks (secret scan + SAST) run on the raw code before anything else. SCA runs at build time when dependencies are resolved. DAST needs a running app. Each check is placed at the earliest possible point where it can provide value — this is the essence of shift-left.
Question 5 of 5 — 🇨🇦 Federal GC Environment
A GC department's Security Assessment & Authorization (SA&A) team asks you to provide "automated security evidence" for their ITSG-33 compliance package. What DevSecOps artifacts would you provide?
✅ Correct! This is a CRITICAL GC interview answer. SA&A teams love automated, continuous evidence because it shows security is built-in, not bolt-on. Your pipeline generates SAST reports (mapped to ITSG-33 SI controls), SCA/SBOM (supply chain security), and DAST reports (application security) automatically on every single build. THIS is what gets federal systems approved faster.
